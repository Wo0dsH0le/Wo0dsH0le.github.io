Windows下Firefox72.0源码编译

概述

虽然 Linux上编译 Firefox很简单，几乎不会遇到问题（也随带提一下），但是 exploit常是针对 Windows环境开发的，并且调试、分析工具也更为顺手。

因此决定踩一踩 Windows上编译 Firefox的坑，记录总结分享一波。

最后还单独编译了 JS引擎（其实这才是初衷……）

PS：本文重在记录编译过程中遇到的问题，Firefox的编译流程请参照官方的编译文档自行了解，不在文中赘述。

问题1：依赖

不管是 Windows还是 Linux，编译特定版本的 Firefox遇到的最大问题，其实是其所需要的各种依赖。

尽管第一步bootstrap.py （Linux）| MozillaBuild：./mach bootstrap（Windows）都是在解决这个问题，但是总会安装不全。

解决方法相同，hg安装上后：

hg clone https://hg.mozilla.org/mozilla-central/

获取其最新的源码，在此基础上执行 ./mach bootstrap

此后会出现如下提示：

Would you like to run a configuration wizard to ensure Mercurial is
optimally configured? (Yn):

别被它委婉的语气迷惑，要选择继续，才能把依赖安装完全。

之后切换到下载的特定源码目录下进行编译./mach build，Linux上还可能遇到：

clang: error: unable to execute command: Killed

这是由于 swap大小不够，扩展即可。

Linux上再设置一下 mozconfig的环境变量：

export MOZCONFIG=$HOME/mozilla/mozconfig-firefox

上述就是 Linux上编译特定版本 Firefox源码会遇到的问题。

Windows上的坑还要继续……

问题2：vs2017

官方编译使用的是 vs2017，因此我的编译环境是：

  • Win7 SP1 x64旗舰版
  • vs2017 专业版，v15.0

Win7上安装 vs2017需要先更新 .NET，甚至官方文档还要求进行系统更新。

但有点无语的是，.NET的更新补丁是在可选里的。Windows系统更新完后，vs2017还是不能安装，我觉得直接更新 .NET即可。

vs2017安装时按官方要求选上 4个要求项即可。

但需要注意的是，安装好后运行一次，第一次运行 vs会有一些配置工作。

问题3：MozillaBuild

默认安装在 c盘下，创建快捷方式 c:\mozilla-build\start-shell.bat到桌面。

Rust-1

$ ./mach bootstrap
mach bootstrap is not fully implemented in MozillaBuild

Could not find a Rust compiler.
Will try to install Rust.
Downloading rustup-init... Ok
Running rustup-init...
error: Found argument '--component' which wasn't expected, or isn't valid in this context

解决：

之前 Linux上遇到这个错误，用 mozilla-certral代码即可安装此依赖。

Windows上需要手动安装 Rust。

Rust-2

 0:10.80 ERROR: Cannot compile for x86_64-pc-mingw32 with C:\Users\pc\.rustup\to
olchains\stable-i686-pc-windows-msvc\bin\rustc.exe
 0:10.80 The target may be unsupported, or you may not have
 0:10.80 a rust std library for that target installed. Try:
 0:10.80   rustup target add x86_64-pc-windows-msvc

解决：

rustup target add x86_64-pc-windows-msvc

Rust-3

0:04.63 ERROR: The rust compiler host (i686-pc-windows-msvc) is not suitable for the configure host (x86_64-pc-mingw32/x86_64-pc-windows-msvc).

解决：

rustup toolchain install stable-x86_64-pc-windows-msvc

PS：后面两个问题看似是 Rust安装是与系统位数没对应导致的，但其实我是安装 64位下的默认方式安装的……

问题4：mozconfig

这一部分参照官网文档写即可，我用的是最简版本：

mk_add_options MOZ_OBJDIR=@TOPSRCDIR@/obj-@CONFIG_GUESS@
ac_add_options --enable-application=browser
ac_add_options --enable-debug

但注意 Windows 64位下，需要加上：

ac_add_options --target=x86_64-pc-mingw32
ac_add_options --host=x86_64-pc-mingw32

mozconfig没有文件后缀名，置于源码目录下即可

windows上我没有设置 mozconfig相关的环境变量，但是在编译过程中我们自定义的 mozconfig还是可以正常找到。

PS：其实这一步中我最关心的是ac_add_options --enable-jitspew这个编译项。但在官网、Google上都没都搜到，而对于探索 Firefox的 JIT引擎很有帮助，会输出很多中间信息。

这一点待后面单独编译 JS引擎时解释。

问题5：Compile

PSandboxTestingParent.h缺失

21:45.97 c:/firefox-72.0/security/sandbox/common/test/SandboxTestingParent.h(10,
10): fatal error: 'mozilla/PSandboxTestingParent.h' file not found

编译过程中出现出现最多的问题，是 PSandboxTestingParent.h源文件的缺失。

我下载了多个版本的 Firefox源码，最早到了 2018年发布的 60.0版本，也都没有此文件（本应在 js/src/vm下）。

解决：

到\security\sandbox\common下查看 moz.build代码。

可以发现用到此文件的部分，都是和测试相关的，因此可以选择注释掉 moz.build中相关代码。

相同的注释源码处理，后续会用在两部分源文件中：1.Firefox本身的源码；2.编译时生成的源码

本身的源码：

问题6：mozconfig

Firefox的 JS引擎需要单独编译得到，按照官网文档即可：

cd js/src
# This name should end with "_DBG.OBJ" to make the version control system ignore it.
mkdir build_DBG.OBJ
cd build_DBG.OBJ
/bin/sh ../configure.in --enable-debug --disable-optimize --enable-jitspew
# Use "mozmake" on Windows
make

但我最关心的是 ac_add_options --enable-jitspew。

充分利用编译器自身提供的调试信息输出，这经常在高质量的漏洞分析文章中出现。

如何学会在 Firefox上获得这种信息，才是我源码编译的初衷，而这些调试相关的编译配置项，竟然查不到相关资料。

一番折腾而发现，在js/src下 configure.in --help就可以列出各种配置项及说明。同理，返回 Firefox源码根目录也可（弥补了官网 mozconfig文档的简略）。

其实 mozconfig和在 configure.in后配置参数效果一样，并且configure.in运行后还会检查各种依赖，在 build输出文件下生成 vs工程。
